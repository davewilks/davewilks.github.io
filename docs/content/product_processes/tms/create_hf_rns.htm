Create Hotfix Release Notes

Summary:

Describes how to deliver Hotfix Release Notes (HF RNs) every two weeks for the Willis Towers Watson Talent Management Software project. Also defines process to update Help Center topics.

Full Description:

TComm was asked to write the HF RNs in a process improvement request from Lynn Prescott to make the RNs more readable (less techie) for external client use. See the 16 February 2016 Doc Update Meeting notes for reference.

For about six months, a TComm writer created the HF RNs every two weeks for the HF Team in Manila. As of mid-July 2016, the new process is that the hotfix tickets are part of the regular US and PH development sprints. For writers, the process involves creating a short summary and resolution for each item, adding these to a new HF RNs file using the HF RNs template, and then going through one peer review and one functional review with the Product Owner (PO). TComm then discusses the final deliverable with the PO. TComm usually makes any changes on the last delivery day and creates the final Word file (with both versions in it) and attaches to the TComm ticket. TComm then sends an email to the PM/PO who sends out an internal HF email with the latest RNs file and a table of the tickets in the file. If the PM/PO make any final changes, they attach another final file to the TComm ticket. After the email is sent, TComm creates the two different Word and PDF files, attaches them to the TComm ticket, and uploads the HF RNs files to the TMS Supporting Docs page.

The hotfix tickets sometimes have an impact (updates, new screenshots) on the TMS Help Center topics. The step to manage those is at the end of the following RNs process.

Tips

  • TComm has about two weeks to finalize the RNs with the PO. The US TComm writer then has another day (Wed Manila time = Tues Denver time) to manage any last-minute ticket changes to the HF RNs. Sometimes the PO must write the last-minute tickets.
  • The HF tickets are in the dev sprints. See the latest TMS board and use the HF filter.
    • Week 1: Wed-Tuesday
    • Week 2: Wed-Tues (but send HF RNs email on the second Friday)
  • Do not clone JIRA tickets as they then include links and attached files.
  • Write a one-to-two-sentence summary and a one-to-three sentence resolution for each HF ticket. The summary explains what the user sees (the problem). The resolution explains what the user sees after the problem has been corrected (the corrected way the software should work) and a brief configuration explanation, if necessary, for the correct view.
  • Use this list to check the HF RN drafts for writing style errors and inconsistencies: Editing Checklist for HF RN Drafts.
  • Use this checklist to perform periodic clean up tasks on the TMS Hotfix Release Notes before sending them to product owners for final approval: Final Draft Checklist for TMS HF RN.
  • Do not include images in the HF RNs.
  • You may not use every table in the HF RNs for a release.
  • Here is a sample TComm Ticket: http://jira.ehr.com/browse/TCOMM-1870.
  • Use this TMS build to review tickets if the TRSS ticket doesn't list a build to use:
    https://tms-productqa.internal.towerswatson.com/Admin/    . Ask the PO for your own login.
  • You can also use Client C to check older builds: https://trt/ClientC/Admin/wwc/Login.aspx.
  • The table below contains all of the HF boards and ticket samples. Here is the PM's TMS Hot Fix Process Visio and Hot Fix Process Details.
  • The PO (US or PH) adds labels to the HF tickets so you know immediately if that ticket is in the HF RNs. The labels are ReleaseNotes and the HF label such as 8.5HF-1.
  • The Component field in the ticket tells you which table in the RNs to put the ticket in.
  • If you review a ticket that does not have clear instructions on reproducing the problem and solution, alert the PO/PM as the repro steps should show what should happen (formerly acceptance criteria but more info now).
  • The HF RNs are also included in the Client Portal. The Client Delivery team copies the files on the Confluence > TRSS Supporting Documents page and then removes the JIRA ticket links before posting the RNs to the portal for external client use.

Weeks 1 & 2 Week 2: Mon TComm
  1. Every day review the tickets in the boards: (4-20 hours over two weeks)

    • TMS HF Dashboard

      First Filter: Status of the current Hot Fix (ticket number and current status)

      Second Filter: Status of the HF tickets in the release backlog (if a ticket is still Open, Ready for Dev, Blocked, etc).

  2. Use TCOMM or TRSS JIRA tickets to discuss work with PO so that TComm can work on tickets. (FYI: Rarely add comments to TRSS ticket. Mostly add to TCOMM tickets and ask questions in refinement and planning HF meetings.)

    Each writer creates one Word file (v#) to create draft for first review for versions.
    • In tickets that need to go into the RNs, the POs create labels for the versions (e.g., 8.5_HF3) and ReleaseNotes.

    • Write the summary and resolution for each ticket.

    • Attach your interim files to the HF RNs ticket so no work is lost if another TComm writer has to pick up your work or review it.
    • Check in the files every day.

    • Assign PO to ticket.

Every week: Tues & Thurs

Tues & Thurs

 TComm

Attend weekly HF meetings:

  • Contact PO/PM to add you to the email notice for the weekly meetings which discuss current hotfix releases, upcoming releases, and ticket status.
  • Add top ticket list to TCOMM HF JIRA ticket to keep track of possible tickets.
  • Attend demos in meetings. Taking notes in these meetings helps with the write-up of the summary and resolution in the HF tickets. POs records HF meetings.
Weeks 1 & 2: Wed-Wed Wed-Wed TComm/PO

Go through the revision process:

  1. Use TCOMM ticket to review HF RNs.
  2. Rename review file for each new round of edits and attach to TCOMM ticket.
  3. TComm writers review edits, make changes, and resolve open items.
  4. TComm writers attach any new tickets and new files for POs review.
Week 2: Tues-Wed Wed TComm

Create final files of HF RNs:

  1. Resolve and enter all edits. Change any content
  2. Use this checklist to perform periodic clean up tasks on the TMS Hotfix Release Notes before sending them to product owners for final approval: Final Draft Checklist for TMS HF RN.
  3. Discuss the final files with the PO. If there are late changes, work with the PO to make final changes and determine who will create the final file.
  4. Attach the final file to the ticket. It includes both versions.
  5. Email PM/PO that the final files are attached to the ticket.
  6. Reassign TComm ticket to writer.
  7. After email comes out, create Word/PDF files for each version and attach to ticket.
  8. Attach Word and PDF copies to TMS Supporting Documentation Confluence page.
Week 2: Wed Wed PM

Email final files:

  1. PM updates distribution list.
  2. PM sends out HF RNs email with one PDF file attached. Includes list of new issues in email.
Week 2: Wed Wed TComm

In the TComm ticket that impacts the HCTs, go through the RNs and tickets to see if any HF changes impact the TMS HCTs.

  • If there are no impacts, close the ticket.
  • If there are impacts, create a TComm ticket and add it to the top of the backlog in the Tech Comm: TRSS board. Assign to a TComm sprint to work on. Go through the regular ticket review process with the PO.
Week 2: Wed Wed TComm
  1. Verify all work is done and make sure to close the two TComm tickets.
  2. Lastly, close the sprint.
