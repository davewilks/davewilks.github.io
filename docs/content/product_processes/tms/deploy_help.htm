Summary

Describes the process for deploying help to QA/UAT site for T-Comm testing and cleanup and then to Production site for client release.

NOTE Begin this process about two weeks before the release date. In an ideal world, topic updates should mostly stop and focused testing should start about one week before the date.

Folder Setup and Tickets

About two weeks before release date:

  1. In the QA and Prod server file systems, create a new folder for the release version, e.g.:
    • In QA: \\trtiis-na2-02q\deploy\tr-help\8.5
    • In Production: \\trtiis-na1-06\deploy\TRHELP\8_5
  2. Into each of the new folders, copy the following file from the parent folder:

    HelpReceive.ashx

  3. Create a DevOps ticket for QA to update the following configurations in ehrAppConfig.config:
    • TRT.HelpAppVersion="<version_number>" (from step 1, above)
    • TRT.HelpUrlContentPath="https://staging6.ehr.com/tr-help/<version_folder>/default.htm"

  4. For example, see http://jira.ehr.com/browse/RMS-28875.

    Create a DevOps ticket for Prod to update the following configurations:

    • TRT.HelpAppVersion="version_number"
    • TRT.HelpUrlContentPath="https://tr-help.ehr.com/<version_folder>/default.htm"

    For example, see http://jira.ehr.com/browse/DEVOPS-290.

    IMPORTANT: Add to this ticket instructions to set up internet security configuration in web.config files for BOTH QA and Prod environments root directories:

    • \\trtiis-na2-02q\deploy\tr-help\web.config
    • \\trtiis-na1-06\deploy\TRHELP\web.config

    Per instructions in confluence. .

  5. Create an RMS or DevOps ticket to set Client C to point to the released Production help.

    For example, see the comment for settings in RMS-21933.

    The only config settings to change are:

    • TRT.HelpAppVersion="8.x" [use real number]
    • TRT.HelpUrlContentPath="https://tr-help.ehr.com/8.x/default.htm"

    We need client c to point to production for testing security features later--see Post-release Tasks.

NOTE Ensure the DevOps tickets are resolved before the release date.

Help Center Prep

No later than two weeks before release, enter "Prelease schedule..." planning info at the top of the Help Issues spreadsheet and start going through the sheet entries to identify and fix issues that still need fixing. Carry out prerelease checks if not too soon -- e.g. Analyzer (see color legend at the bottom of the sheet).

A week before release, start paging through the help to find new issues. Enter in the spreadsheet for assessing priority, or fix and log there for verification in TRSS - Reviewers site (better for linking to from the sheet--direct links to QA may be blocked).

QA and Production Deployment

About two weeks before release, prepare and verify build in QA environment. A week before release, prepare and verify build in Production.

Prepare and Test QA Build

Two weeks before release:

  1. Ensure that the TRSS - Reviews TOC is up-to-date with all new topics and that the order and hierarchies are correct.
  2. With the Reviews TOC open, check out the TRSS - QA TOC.
  3. Save the Reviews TOC (File > Save > Save As) under same file name used for the QA TOC. (Select all files in the dialog box to see and select the QA TOC file name.)
  4. In Projects > Destinations > QA_Environment, change the Directory box to the file location you created in step 1 under Folder Setup and Tickets.

    Also change the View URL box to use the release version number.

  5. In Projects > Targets, right-click QA_Environment and click Publish.

    (Or in the Projects menu ribbon, click Publish Primary (the drop-down not the icon) and select Publish "TRSSQA_Environment".)

  6. Go to the View URL -- http://staging6.ehr.com/tr-help/8.x/default.htm -- and smoke test (verify landing page has correct publish time stamp/version number and click through a few pages and TOC links). Enter problems in google Help Issues spreadsheet.

    NOTE If the URL results in no-access message, use the back-door through the file system: see step 1 under Folders... above.

  7. Repeat the previous two steps iteratively as content changes are made until the help is ready for release. (You can set up a different batch schedule for this if you like--just be sure to turn it off (clear Enable box in the batch scheduler dialog) after the release.)

Prepare and Test Production Build

One week before release, follow the same steps used for QA, above, using the production TOC, Destination, and Target files, and the production site for publishing and smoke test.

Post-release tasks

Soon after the release is posted, do following:

  • Zip your local TRSS1 output production folder; ./TRSS - Prod/ (faster than using published output in Timbuktu), and deposit the zip at http://nateamsites.internal.towerswatson.com/lob/dst/GDTTechComm/TRSS%20Source%20Control
  • If you used a batch schedule for release to QA/Prod, clear the check boxes in the batch target setup--disable the schedule if you're using the BatchTRSSReviews setup.
  • In both the QA and Production Destination files (in Project Organizer), change the release version folder under Directory: Either change to the future version number or surround the number with illegal characters such as { } (brackets.

    This a failsafe measure to prevent interim accidental publication, overwriting the site content--fat fingers can happen!

  • Test client help site access.
    • For QA:
      1. Log on to any QA environment and click Help--ensure that the help welcome page appears and that the Last Published date at the bottom and the version number in the footer are correct.
      2. Switch browsers or dump cookies (I prefer the former).
      3. In the browser location field, try the URL for the help site--ensure the result is this error page:

        The URL is https://staging6.ehr.com/tr-help/<version_folder>/default.htm.

      4. If either of these tests fails, ensure that all the site preparation steps were accomplished and that DevOps tickets were resolved.

      For Production:

      1. Log on to Client C and click Help--ensure that the help welcome page appears and that the Last Published date at the bottom and the version number in the footer are correct.
      2. Continue with the same steps used for QA above, substituting the Prod URL at step 3--https://tr-help.ehr.com/<version_folder>/default.htm.
  • In Google spreadsheet, right click the current release sheet and click Duplicate, then rename the sheet (right-click again) to the next release number and delete fixed items. Right-click again and Move left; repeat until sheet is far left.
  • Have a party!!

Update a Past Release

So what happens if you need to make a small change to a help system after it deploys to a release...well instead of republishing the entire help system, just go into the release directory and change the .htm file. For example, for 8.4, go to \\trtiis-na1-06\deploy\TRHELP\8_4 and change the file with Notepad or some other editor.
